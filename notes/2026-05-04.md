## Background worker
- We need a queue that the API and the worker can share. It should be injected as Singleton, since they both need to be looking at the same queue instance.
- We will use `System.Threading.Channels.Channel<T>` - a thread-safe producer/consumer queue built into the framework.
- I can use an unbounded channel - no limit on the number of messages it can queue up.
- I really do not like just using things like `Order` without declaring where they are coming from. The lack of imports feels so yucky to me.
  - Apparently i could add explicit `using` statements if it makes me feel better. I will try to follow the .NET paradigm and accept the implicit imports.
  - Actually you *must* add a using statement for `Channel`: `using System.Threading.Channels;` since it's not in the default implicit usings.
- `Console.WriteLine()` will write to the console.
- The worker needs the queue to be dependency injected from the program.
- `channel.Writer.WriteAsync(order)` in the program writes the order to the queue.
- `_channel.Reader.ReadAllAsync(stoppingToken)` in the worker reads everything off the queue.
- No need for a Task.Delay() - it's all handled by `await foreach (var order in _channel.Reader.ReadAllAsync(stoppingToken))`
- Insight: The program is still going to do what it did with the order - AND we are ALSO enqueuing it for background processing.
- Create the order first, then enqueue it. If it fails to create, nothing goes on the queue.
- `await channel.Writer.WriteAsync(order);` - the `await` is important (like TypeScript)
- The router just wants to store the order as fast as possible and respond with 201 Created.
- The background worker is intended for things like sending a confirmation email, reserving inventory, notifying a delivery service.
- Background worker has nothing to do with persisting the order. We'll get to adding a database later.

# General notes
- A `channel` is the .NET abstraction for a producer/consumer pipe in memory. A `queue` can cross process boundaries.
