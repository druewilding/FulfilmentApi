## Domain Events
- Domain events are an internal record that something significant happened inside the aggregate.
- BEFORE broadcasting anything to the outside world.
- So they happen inside the domain model and are seen by the same process.
- **aggregate** is a cluster of related objects, e.g. Order and OrderItem.
- **aggregate root** is the entry point into that cluster. Order is the root.
- What should an `IDomainEvent` look like? It's an interface with:
  - timestamp (`DateTimeOffset OccurredAt` is typical)
- Then you get classes that use the interface:
  - `OrderPlaced` - with `Guid OrderId`
  - `OrderConfirmed` - with `Guid OrderId`
- Domain events added at the right times and asserted in the domain tests.

## Real message publishing with RabbitMQ
- Use docker-compose to spin up a RabbitMQ docker image
- Management UI is available at http://localhost:15672/ (username and password defined in docker-compose file)
- RabbitMQ runs on Erlang - the lightweight process and "let it crash" philosophy make it great for a message broker.
- Add MassTransit into the project, a library to help connecting to RabbitMQ.
- Port 5672 is the standard port for AMQP (Advanced Message Queuing Protocol) (like 443 is standard for HTTPS)
- MassTransit v9 needs a license, so I'm downgrading to v8.
- Added MassTransit connection to RabbitMQ in the `Program.cs` and i see it connecting in the Management UI 😊
- MassTransit is dependency injected into the container, so `IPublishEndpoint` is available anywhere.
- The router is as thin as possible, and OrderService is where the message should get published.
- One problem: `IOrderService.CreateOrder` is currently synchronous. We need to make it async.
  - Marking the return type as `Task<Order>` automatically makes it async-compatible (can use `await`).
  - We use `return order;` as before - the compiler wraps it in a Task.

## DI insight
- You don't need to explicitly pass things around from the Program.
- OrderService is registered with `builder.Services.AddSingleton<IOrderService, OrderService>();`
- `IPublishEndpoint` is made available with `builder.Services.AddMassTransit`
- When the DI container sees that OrderService needs an IPublishEndpoint, it injects it when constructing the service.
- You just declare what you need in the constructor, and .NET figures out the wiring.

## DI lifetime conflict
- A common .NET DI mistake!
- `IPublishEndpoint` is **Scoped** - one per HTTP request.
- OrderService has been registered as **Singleton** - one for the whole app lifetime.
- A singleton cannot hold on to a scoped service because it would never get released.
- We made OrderService singleton so that we could store orders in memory.
- Now is the time to make an `OrderRepository` class, which *can* be dependency injected as singleton.

## Test mocking with NSubstitute
- The tests also need to inject an `IPublishEndpoint` - we can do this with a mock.
- `var publishEndpoint = Substitute.For<IPublishEndpoint>();`
- Since we mock instead of actually connecting OrderService to MassTransit, we do not see new messages enter the RabbitMQ.
- This keeps tests nice and fast.
