# Saturday 2 May 2026

- Installed dotnet using Homebrew, got .NET 10 - decided that will be ok.
- .NET 10 creates solution file as .slnx (.sln is legacy)
- `webapi` is a runnable application, like an Express app.
- `classlib` is just a library - just code, doesn't run on its own. Sort of like an npm package.
- **Architectural rule**: the Domain layer must stay pure, just C# classes with no infrastructure references. `classlib` enforces that rule.
- A lot of scaffolding was created that we will replace later.
- `dotnet new gitignore` generates a useful .gitignore file.
- There are commands to add projects to the solution file.
- References talk about dependency direction. **Dependencies point inward.** Infrastructure needs to know about Domain. Domain knows about nothing. The API knows about both.
- .NET generates OpenAPI spec at runtime directly from the code. I'll be able to see it at http://localhost:5142/openapi/v1.json
- FulfilmentApi.http is not auto-generated. I should keep it up to date, it's like a Postman collection.
- A `record` is like a `class` but for **immutable data** and with **structural equality**.
- Reference counter is a benefit of a typed language with a good IDE.
- Two-phase pattern: **build** (`builder.*`) - register and configure services, and **pipeline** (`app.*`) - define routes, middleware, run the app.
- `dotnet run --project src/FulfilmentApi` builds and runs the app.
- **REST Client** extension allows me to send requests as defined in the FulfilmentApi.http file.
- Automatic OpenAPI generation is *amazing*!!
