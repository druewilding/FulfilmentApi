## Data persistence
- Started by storing the orders in a dictionary so that we can look them up again
- `TryGetValue` is a good way of looking up in a dictionary
- Now we need a **service layer** (interface)
- .NET usually indents with 4 spaces
- If you update code return type you must update the interface to match (obvious, but i missed it)
- `builder.Services.AddScoped<IOrderService, OrderService>();` - this injects one instance per HTTP request. So my order was created, but then thrown away.
- Change to `AddSingleton` to persist in-memory store across requests.
- Full picture:
  - `AddSingleton` - one instance for the lifetime of the app
  - `AddScoped` - one instance per HTTP request
  - `AddTransient` - one instance per injection

## Logging middleware
- Middleware is a chain of functions that every request passes thorough _before_ it gets to the route handler.
- `app.UseHttpsRedirection()` is middleware, as is `app.MapOpenApi()`.
- .NET has `app.UseHttpLogging()` built in, but we're going to write it from scratch.
- `app.UseMiddleware<RequestLoggingMiddleware>();` - Middleware needs to be explicitly registered by the program - it doesn't just find it automatically.
- Middleware can also work on the outgoing response, after being handled by the route.

## Unit tests
- Test projects also need the reference to the web project, so that they can see the code.
- In tests you are in control, you are testing the exact thing. Therefore it's correct to use `new OrderService()` rather than dependency injection.
- Interfaces, classes and records need to be `public` for the test to be able to see them (default is `internal`).
- No need to add integration tests yet, as the routes are merely a thin wrapper over `OrderService`. Will make more sense when we add database, message bus, midleware interactions (test full stack).
